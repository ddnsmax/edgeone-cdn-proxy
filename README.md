这是一套可以部署在 EdgeOne Pages 上的反代系统。

它不仅仅是一个简单的反代，更是一套**“前台路由调度 + 后台重载下载”**的双轨制矩阵系统。它可以把你的访问入口网站放到 EdgeOne 的边缘节点上，让用户先访问离自己更近的节点，同时将大文件和音视频的下载流量巧妙地分发到 Cloudflare 节点。实际体验通常会体现在以下几个方面：

- 访问入口更统一，多个域名可以集中管理
- 海外、跨地区访问时，打开速度通常会更稳一些
- **独创大文件分流机制，网页/静态资源走 EdgeOne，大文件/媒体流自动转交 Cloudflare 处理**
- 可以把一个普通站点，做成更像 CDN 的访问方式
- 后台可直接添加、修改、开启、关闭反代规则
- 支持伪装首页、后台入口、多个反代域名管理，**支持配置多个下载节点进行随机负载均衡**

这不是万能提速工具，最终效果仍然会受到源站质量、线路、文件大小、访问地区和平台策略影响。但在大多数正常场景下，它确实可以让网站访问和下载体验更顺一些，同时极大节省高昂的 CDN 带宽成本。

## 核心工作原理（双平台流量分离）

为了规避 EdgeOne Serverless 平台对单次请求时间和流式大文件的限制，同时有效利用 Cloudflare 海量的免费出站带宽，本系统采用了**流量分流**策略：

1. **常规请求**：HTML 网页、小型 JSON、普通 API 均由 EdgeOne 节点直接反代，利用 EdgeOne 优秀的区域连通性提升网页首屏加载和操作体验。
2. **重载请求**：当系统检测到用户正在请求大文件（>20MB）、附件或音视频媒体流（如 mp4、mp3 等）时，EdgeOne 节点**不会**亲自拉取数据。它会生成一个带有动态加密鉴权签名（Sign）和有效期（Expire）的临时专属链接，向用户下发 302 重定向。
3. **下载分担**：用户的真实下载请求会被静默转移到你部署好的 Cloudflare 下载节点上，由 Cloudflare 来承担大文件下载的沉重带宽压力。

## ⚠️ 核心安全必读：密钥 (SECRET_KEY) 配置

本系统采用“主控+节点”的通信架构。为了防止您的 Cloudflare 节点被他人恶意蹭网盗刷流量，同时确保大文件分流功能握手成功，**您必须在 EdgeOne 和 Cloudflare 两端的代码中，设置完全一致的通讯密钥！**

**需要修改密钥的两个文件位置：**

1. **EdgeOne 主控端代码**：
   在源码的 `edge-functions/_shared.js`文件顶部第一行，找到：
   `const SECRET_KEY = 'YOUR_CUSTOM_PASSWORD_HERE';`
   请将引号内的默认值，修改为您自己的复杂随机密码。

2. **Cloudflare 下载端代码**：
   在源码的 `worker.js` 文件顶部第一行，找到相同的一行代码：
   `const SECRET_KEY = 'YOUR_CUSTOM_PASSWORD_HERE';`
   请将其修改成与主控端**完全一模一样**的值。

<p><strong><span style="color:red;">强烈警告：如果您不修改此密钥，将面临极大的流量被盗刷风险；如果两端密钥不一致，大文件下载将被直接拦截并报错（Forbidden）！</span></strong></p>

## KV 怎么填

这个项目需要绑定一个 KV，用来保存初始化信息、后台设置和反代配置。

请按下面填写：

- 类型：KV
- 变量名：`PROXY_KV`

## 部署流程

本系统需要分别在 EdgeOne 和 Cloudflare 两端进行部署，以实现完整体验。

### 第一阶段：部署主控分发端（EdgeOne Pages）
1. Fork 本项目到自己的 GitHub 仓库。
2. 进入 EdgeOne Pages，选择从 Git 仓库部署。
3. 选中你 Fork 后的仓库，创建项目。
4. 先在项目里绑定自定义域名。
5. 按平台提示完成解析，等待证书生效。
6. 在 Functions 绑定里创建并绑定一个 KV，变量名填写为 `PROXY_KV`。
7. 保存后重新部署项目。
8. 用你刚绑定好的自定义域名访问网站。首次打开会进入初始化界面，按提示完成初始化。

### 第二阶段：部署真实下载端（Cloudflare）
9. 获取下载端源码：访问 [cloudflare-pages-download](https://github.com/ddnsmax/cloudflare-pages-download)。
10. 部署下载节点（根据你的操作习惯，以下两平台二选一即可）：
    * **方式一：使用 Cloudflare Pages 部署（推荐，最省事）**
      在 Cloudflare 后台新建 Pages 项目，选择“连接到 Git”，直接拉取上述仓库的代码，平台会自动完成部署。
    * **方式二：使用 Cloudflare Workers 部署（直接粘贴代码）**
      在 Cloudflare 后台新建一个 Worker。在上述仓库源码中找到 `worker.js` 文件，将其中的代码全部复制，粘贴并覆盖到你的 Worker 代码编辑器中，保存并部署即可。
11. 部署完成后，在 Cloudflare 后台为你刚建好的 Pages 或 Worker 绑定一个访问域名，这个域名就是你的“节点地址”。
12. **进阶提示**：你可以利用多个不同的域名或账号，在 Cloudflare 部署无数个这样的下载节点。

### 第三阶段：后台系统对接
13. 使用初始化时设置的账号登录 EdgeOne 主控端后台。
14. 切换到**“下载节点”**配置页，点击新增。
15. 填入你刚才在 Cloudflare 部署好的“节点地址”以及方便自己识别的“备注”。
16. 点击保存。系统会自动测试联通性和鉴权密钥匹配，测试通过后即刻生效。（添加多个节点后，大文件下载时系统会自动随机分配一个节点进行分流）。
17. 最后，切换到“反代配置”添加你的源站，即可开始享受完整的反代分发体验。

## 这个系统适合怎么用

你可以把它当成一个轻量的反代入口来用。

常见用法包括：

- 给下载站、导航站、工具站做一个更稳的访问入口
- 把源站放在后面，用自己的域名统一对外
- 给静态资源、安装包、图片、脚本、更新文件做分发入口
- 做多个站点、多条线路、多域名的统一管理
- 把网站入口放到 EdgeOne 节点上，尽量改善跨地区访问体验

如果你的源站本身很慢、带宽很小、响应不稳定，那么它不能把低质量源站变成高性能服务器；但如果你的源站本身可用，这套系统通常可以把访问入口、下载入口和分发方式整理得更好用。
